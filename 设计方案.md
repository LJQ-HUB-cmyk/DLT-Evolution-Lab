# dlt-evolution-lab 独立项目方案

## Summary
- 在 d:\cursor_git\dlt-evolution-lab 新建独立项目，采用 React + Vite + TypeScript 前端、FastAPI + Python 后端、JSON 文件持久化、Windows 任务计划 + 本地服务 自动任务架构。
- 历史数据只接官方来源，使用 中国体彩网历史走势页、中国体彩网开奖页、中国体彩网游戏规则 做同步、校验和规则版本管理。
- 系统采用“正式预测 + 实验预测”双轨制：每个目标期只保留 1 份正式发布结果作为基准；每次点击都重新计算 1 份实验结果，并立即做漂移检测、自动降权、自动优化入队。
- 模型采用“透明特征打分 + 位置概率校准 + 组合搜索 + 滚动回测 + 自动调参”的可解释体系；大模型只参与复盘和策略建议，不直接改生产权重。
- 业务目标锁定一等奖和二等奖，但系统只做量化优化与复盘迭代，不在界面或逻辑中宣称“必中奖”。

## Architecture & Interfaces
- 项目目录固定为：apps/web 前端、apps/api 后端、jobs 自动任务、data/raw 官方原始数据、data/normalized 标准化历史数据、storage 预测与模型 JSON、artifacts/backtests 回测与校准产物。
- 前端页面固定为两栏：左侧 1/3 展示每期大乐透历史开奖号码，从新到旧纵向滚动；右侧上半区展示数据状态、模型说明、位置概率、方案1/方案2结果；右侧下半区展示每次实验计算、漂移记录、自动优化记录、开奖后复盘和回测图表。
- 视觉风格采用“官方彩票票面 + 数据实验室”混合设计：米白纸感背景、深蓝信息面板、前区红球后区蓝球、金属描边卡片、清晰图表和大面积留白；桌面端严格左右布局，移动端折叠为上下布局。
- 核心 API 固定为：GET /api/issues、POST /api/sync、GET /api/analysis/{targetIssue}、POST /api/publish/{targetIssue}、POST /api/predict/{targetIssue}、POST /api/anchor/recompute、POST /api/postmortem/{issue}、POST /api/optimize、GET /api/models、GET /api/runs。
- 核心持久化文件固定为：issues.json、predictions.json、anchor_ticket.json、model_registry.json、optimization_runs.json、postmortems.json、scheduler_logs.json。
- 对外类型固定为：DrawIssue、Ticket、PredictionRun、OfficialPrediction、DriftReport、ModelVersion、BacktestReport、PostmortemReport、AnchorTicketState。

## Model & Iteration Logic
- 预测引擎使用透明特征，不走黑箱直选。前区 01-35、后区 01-12 每个号码都计算固定特征：最近 10/30/50/100 期频次、当前遗漏、EWMA 热度、邻号/重号、尾号、区间占位、和值贡献、跨度贡献、冷热标签、上期干扰项。
- 位置分析按升序后的前区 1-5 位和后区 1-2 位建模，输出“每个位置的每个号码评分”和校准后的命中率评分；页面必须展示 Top N 号码球、评分、主要贡献因子和历史回测命中情况。
- 结构层把你要求的大小比、奇偶比、三区均衡、冷热搭配、和值、跨度、连号、同尾、非极端形态全部参数化，不写死单一规则，而是作为可学习权重参与总分。
- 方案1固定输出 5 组：1 组永久跟买号、4 组结构化模型号。永久跟买号单独优化，目标函数偏向“跨窗口稳定性 + 高奖相关回测表现”；只有点“重算跟买号”才更新。
- 方案2固定输出 5 组轻玄学号：仍从真实历史高分池中取数，但降低硬约束强度，只做轻结构美化，禁止全奇全偶、全大全小、极端连号；同一模型版本和同一 seed 下必须完全可复现。
- 正式预测由当前冠军模型生成并冻结，只用于该期对照、开奖复盘和后续模型评价；任何优化都不能回写已发布的正式结果。
- 每次点击“实验计算”都会生成新的 PredictionRun。系统立刻计算 漂移分，组成固定为：位置差异、号码集合差异、结构差异、综合分差异、方案级重叠差异。
- 若实验结果与正式结果或近 N 次实验结果漂移超过阈值，系统自动降低导致漂移的特征权重或模型信用分，把该模型标记为 unstable，并自动触发一次优化任务。
- 自动优化只作用于候选模型，不直接替换冠军模型。候选模型必须先通过 walk-forward rolling backtest，且满足“高奖加权分提升、稳定性提升、校准误差下降”三项条件，才允许晋升为新的冠军模型。
- 优化目标函数固定为“奖级加权回报 + 位置校准 + 稳定性奖励 - 漂移惩罚 - 过拟合惩罚”。一等奖 5+2 权重最高，二等奖 5+1 次高，其余奖级按重要性递减，只作辅助约束。
- 优化器使用 Optuna 搜索：特征权重、窗口长度、冷热阈值、结构惩罚强度、beam search 宽度、多样性系数、方案2 seed 策略、永久跟买号稳定性参数。
- LLM 只做复盘辅助。输入是结构化统计 JSON，输出是错误归因、特征建议、搜索空间建议和自然语言解释；所有 LLM 建议都必须先转成 candidate experiment，再经回测验证后才能生效。

## Automation
- 自动任务由 Windows 任务计划 负责兜底拉起，本地服务负责执行任务和提供 API。
- 每天 09:00 自动执行一次：同步官方数据、检查是否有漏期、刷新全量回测摘要、更新位置概率缓存。
- 每周一、三、六 20:30 自动检查下一目标期；若尚未发布正式预测，则自动生成正式结果并持久化。
- 每周一、三、六 21:45 起自动轮询官方开奖；一旦检测到新期开奖，立即回填真实结果、计算命中和奖级、生成复盘、触发优化、评估候选模型是否晋升。
- 用户手动点击实验计算时，立即生成实验 run；若发生高漂移，则额外追加一次优化任务。
- 所有自动任务都必须记录输入快照、历史哈希、模型版本、耗时、结果摘要和失败原因，并在页面下半区完整展示。

## Test Plan
- 同一目标期发布正式预测后不可变；连续手动实验 20 次必须都生成独立 run，并正确记录漂移、降权和优化入队。
- 同一历史快照、同一模型版本、同一 seed 的实验 run 必须产出完全一致结果；只有模型版本或 seed 变化时才允许结果变化。
- 所有预测票必须满足大乐透合法性：前区 5 个不重复、后区 2 个不重复、号码范围正确、球形展示符合官方视觉习惯。
- 位置概率、结构打分、综合排序、组合搜索结果、永久跟买号、方案2结果都必须能在历史回测中复算出同一结论。
- 开奖后复盘必须正确回填命中结构、奖级、加权回报、模型升降级记录、稳定性变化和特征贡献变化。
- 数据同步在网络失败、重复期号、字段缺失、规则版本切换时必须进入降级模式并告警，不得污染历史 JSON。
- 页面必须验证桌面端左右布局、移动端折叠布局、球形样式、图表渲染、滚动历史栏、日志表、复盘详情和模型晋升记录。

## Assumptions
- 项目不复用现有仓库，直接在 d:\cursor_git\dlt-evolution-lab 新建独立全栈项目。
- 存储层首版只用 JSON，不引入数据库；若后续运行记录过大，再升级为 SQLite + JSON 导出，但不影响当前接口设计。
- 大乐透规则和奖级计算按官方规则版本管理，若官方后续再次调整规则，按 issueRange 切换奖级映射，不重写旧期结果。
- 首版 LLM 接口按“可选配置”设计，默认可关闭；关闭后算法系统仍能完整运行、自我回测、自我优化和自动复盘。
