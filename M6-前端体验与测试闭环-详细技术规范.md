# M6 前端体验与测试闭环详细技术规范（执行版）

## 1. 目标与原则
- 目标：交付“可用、可审计、可回归”的生产级前端工作台，并建立端到端测试闭环，支撑一次上线。
- 原则：
  - 信息优先：关键风险信息（漂移、复盘、优化状态）优先可见。
  - 可解释优先：任何预测输出必须可追溯到指标与模型版本。
  - 体验优先：桌面/移动端均需稳定、清晰、不卡顿。
  - 回归优先：改动必须被自动化测试覆盖。

## 2. 范围与非范围
- 范围内：
  - 前端信息架构、页面布局、组件职责、交互规范。
  - API 对接规范、状态管理、错误处理。
  - 性能优化、可观测性、前端测试矩阵。
- 范围外：
  - 新预测算法发明（M3/M4/M5 负责）。

## 3. 技术栈（M6 固定）
- 前端框架：`React 18 + TypeScript + Vite 5`
- 图表：`ECharts`（全站统一，不再二选一）
- 测试：`vitest + @testing-library/react`
- 可选 E2E：`playwright`（M6 必须接入基础冒烟）

## 4. 页面架构（必须）

## 4.1 页面分区
- 左侧（桌面 1/3，移动上层折叠）：
  - `HistoryPane`：历史开奖、目标期定位、筛选。
- 右上（桌面右侧上半）：
  - `DataStatusPanel`：同步状态、数据新鲜度、degraded 告警。
  - `ModelMetaPanel`：模型版本、快照哈希、seed、信用分、漂移等级。
  - `PositionHeatPanel`：位置概率热力图、TopN 候选。
- 右中：
  - `PlanTicketPanel`：方案1/方案2票面与结构解释。
  - `ContributionPanel`：贡献因子分解（特征 -> 分数影响）。
- 右下：
  - `ExperimentRunPanel`：实验 run 列表与对比。
  - `RunLogPanel`：自动任务日志、异常原因、重试状态。
  - `DriftTrendPanel`（新增）：漂移分趋势与告警点。
  - `OptimizationPanel`（新增）：优化任务状态、候选模型与门禁结果。
  - `PostmortemPanel`（新增）：开奖复盘、奖级分布、表现趋势。

## 4.2 响应式规则
- 桌面端（`>=1200px`）：
  - 左右双栏，右侧三段垂直栅格。
- 平板（`768-1199px`）：
  - 左栏折叠为抽屉，右侧两段布局。
- 手机（`<768px`）：
  - 全部面板纵向堆叠，关键状态卡固定顶部。
- 强制要求：
  - 不允许横向滚动条。
  - 关键指标卡（模型版本/漂移等级）始终首屏可见。

## 5. 视觉与交互规范（M6 强化）

## 5.1 设计 token
- 在 `styles.css` 中定义变量（必须）：
  - 色板：纸感底色、深蓝面板、红蓝球色、告警色、成功色。
  - 阴影：卡片阴影、球体阴影、悬浮阴影。
  - 边框：金属质感描边与分割线。
  - 字体层级：标题、正文、注释、数字高亮。

## 5.2 动效规范
- 卡片入场：`120ms` 阶梯淡入。
- 数据刷新：
  - 数字变化：`150ms` 数值过渡。
  - 图表变化：平滑过渡，禁止闪烁重绘。
- 交互反馈：
  - 按钮点击有 loading 与禁用态。
  - 错误态显示可读原因和重试入口。

## 6. 前端数据流与状态管理

## 6.1 API 层（`apps/web/src/lib/api.ts`）
- 统一请求封装：
  - 超时控制（默认 12s）
  - 错误码映射
  - 重试策略（仅幂等 GET，可重试 1 次）
- 契约要求：
  - 与后端 schema 同步，禁止 `any` 漂移。

## 6.2 状态层
- 状态分层：
  - `server state`：issues/models/runs/analysis/postmortems
  - `ui state`：筛选条件、折叠状态、当前选中期号
- 一致性策略：
  - 触发 `predict/publish` 后必须局部刷新相关面板。
  - 不允许全页粗暴刷新导致状态丢失。

## 6.3 错误与降级策略
- `sync degraded`：
  - 顶部状态条显示黄色告警并展示原因。
- `optimize failed`：
  - OptimizationPanel 显示失败摘要和 `run_id`。
- `postmortem missing`：
  - PostmortemPanel 显示“待开奖/待回填”而非空白。

## 7. 性能预算（硬门禁）
- 首屏加载：
  - `FCP < 1.8s`（本地标准机，开发环境除外）
  - `TTI < 2.8s`
- 交互性能：
  - 切换期号后关键卡片刷新 `p95 < 400ms`
  - 图表重绘 `p95 < 250ms`
- 包体积：
  - 主包 gzip `<= 320KB`
  - 单图表组件懒加载后首屏增量 `<= 80KB`

## 8. 测试闭环（必须全量落地）

## 8.1 单元测试（vitest）
- 组件渲染：
  - `PositionHeatPanel`, `PlanTicketPanel`, `ModelMetaPanel`, `PostmortemPanel`
- 关键逻辑：
  - 票面格式化、奖级标签映射、状态色映射。
- API 适配：
  - 错误码映射和降级提示逻辑。

## 8.2 集成测试（testing-library）
- 用户链路：
  - 选择期号 -> 触发实验 -> 查看 run -> 查看漂移 -> 查看复盘。
- 异常链路：
  - API 失败 -> UI 错误提示 -> 重试恢复。

## 8.3 E2E 冒烟（playwright）
- 桌面端冒烟：
  - 首页加载、关键面板可见、触发一次 predict。
- 移动端冒烟：
  - 折叠面板展开、关键状态可见。

## 8.4 回归门禁
- PR 合并前必须通过：
  - `npm run test -- --run`
  - `npm run build`
  - 冒烟 E2E（最少 2 条）

## 9. 验收要求（M6 上线门槛）
- 功能验收：
  - M3/M4/M5 关键业务信息在前端可完整查看和追溯。
  - 漂移、优化、复盘三面板均可用。
  - 桌面与移动端布局符合规范。
- 质量验收：
  - 前端新增代码覆盖率 `>= 85%`。
  - 零阻塞错误（阻断主流程操作）。
  - 关键组件无白屏/无空白状态。
- 性能验收：
  - 满足第 7 节全部预算。
- 测试验收：
  - 单测、集成、E2E 冒烟全部通过。
- 文档验收：
  - 输出 `artifacts/backtests/m6_acceptance_report.md`，记录测试证据和风险。

## 10. 执行顺序（不可跳步）
1. 先补 API 类型与状态层契约。
2. 再实现 M4/M5 新增面板组件。
3. 再做响应式与性能优化。
4. 最后补齐测试矩阵并出验收报告。
