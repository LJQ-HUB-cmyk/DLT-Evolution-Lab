# M5 开奖复盘与自动任务详细技术规范（执行版）

## 1. 目标与原则
- 目标：实现“同步 -> 发布 -> 开奖回填 -> 复盘 -> 优化 -> 晋升评估”的全自动闭环，形成可审计运维流水线。
- 原则：
  - 幂等优先：同任务同输入重复执行不得污染数据。
  - 审计优先：所有自动任务必须有完整日志链路。
  - 失败可恢复：任务失败可重试、可补偿、可定位。
  - 数据不可逆污染：历史真实开奖记录只增不改。

## 2. 范围与非范围
- 范围内：
  - 任务调度编排、任务状态机、失败重试与告警。
  - 开奖回填、奖级计算、复盘报告生成。
  - 自动触发优化与候选晋升评估。
- 范围外：
  - 前端视觉重构（M6）
  - 核心漂移算法改造（M4）

## 3. 技术栈（M5 固定）
- 调度：`Windows Task Scheduler + jobs/scheduler_service.py`
- 后端服务：`FastAPI`
- 数据处理：`pandas`, `numpy`
- 持久化：JSON append-only
- 测试：`pytest`

## 4. 代码结构与职责
- `jobs/register_tasks.ps1`
  - 注册/更新计划任务，保证环境变量和执行参数一致。
- `jobs/scheduler_service.py`
  - 任务入口、任务分发、任务状态管理、失败重试。
- `apps/api/app/services/official_sync_service.py`
  - 官方同步服务（M2 延展）。
- `apps/api/app/services/postmortem_service.py`（新增）
  - 奖级计算、命中分析、复盘报告生成。
- `apps/api/app/services/scheduler_audit_service.py`（新增）
  - 统一日志写入 `scheduler_logs.json` 与告警摘要。
- `apps/api/app/services/automation_pipeline.py`（新增）
  - 编排开奖后自动闭环。
- `storage/scheduler_logs.json`
  - 自动任务审计日志。
- `storage/postmortems.json`
  - 复盘报告持久化。

## 5. 任务编排设计（必须实现）

## 5.1 定时策略（北京时间）
- 每日 `09:00`：`sync_job`
  - 执行 `POST /api/sync`，更新历史数据快照。
- 每周 `周一/周三/周六 20:30`：`publish_check_job`
  - 检查目标期是否已发布；未发布则触发 `POST /api/publish/{issue}`。
- 每周 `周一/周三/周六 21:45` 起每 5 分钟：`draw_poll_job`
  - 轮询官方开奖结果，直到检测到目标期真实开奖。
- 开奖检测成功后链式触发：
  1. `postmortem_job`
  2. `optimize_job`（按门槛触发）
  3. `promotion_eval_job`

## 5.2 任务状态机
- 状态：`queued -> running -> succeeded | failed | skipped | compensated`
- 必填元数据：
  - `task_id`
  - `task_type`
  - `trigger_source`（schedule/manual/retry/chained）
  - `idempotency_key`
  - `snapshot_hash`
  - `model_version`
  - `started_at`, `finished_at`
  - `duration_ms`
  - `result_summary`
  - `error_message`（失败时）

## 5.3 幂等键设计
- `idempotency_key = sha256(task_type + target_issue + snapshot_hash + model_version + date_bucket)`
- 幂等规则：
  - 同 key 成功过：后续同 key 任务标记 `skipped`。
  - 同 key 失败过：可重试，但必须新增 `attempt_no`。

## 6. 开奖回填与奖级算法（硬要求）

## 6.1 回填流程
1. 从官方数据提取 `draw_issue`, `front`, `back`, `draw_date`。
2. 校验字段合法性（号码范围、长度、去重、升序）。
3. 若该期已存在真实开奖且内容一致：幂等返回。
4. 若该期存在但内容不一致：标记 `data_conflict`，进入人工确认队列，禁止覆盖。
5. 写入 `storage/issues.json` 与标准化文件，触发复盘。

## 6.2 命中计算
- 对每张票计算：
  - `front_hits = |ticket.front ∩ draw.front|`
  - `back_hits = |ticket.back ∩ draw.back|`
- 同步计算结构命中：
  - 奇偶结构匹配
  - 分区结构匹配
  - 和值区间命中
  - 跨度区间命中

## 6.3 奖级映射（默认大乐透规则）
- `front_hits=5, back_hits=2 -> prize_level=1`
- `front_hits=5, back_hits=1 -> prize_level=2`
- `front_hits=5, back_hits=0 -> prize_level=3`
- `front_hits=4, back_hits=2 -> prize_level=4`
- `front_hits=4, back_hits=1 -> prize_level=5`
- `front_hits=3, back_hits=2 -> prize_level=6`
- `front_hits=4, back_hits=0 -> prize_level=7`
- `front_hits=3, back_hits=1 or front_hits=2, back_hits=2 -> prize_level=8`
- `front_hits=3, back_hits=0 or front_hits=2, back_hits=1 or front_hits=1, back_hits=2 or front_hits=0, back_hits=2 -> prize_level=9`
- 其他组合：`no_prize`

## 6.4 复盘评分（Postmortem Score）
- 对单次预测 run 计算：
  - `prize_score`：按奖级加权（1级最高，9级最低，未中奖为 0）。
  - `structure_score`：结构命中得分。
  - `stability_score`：与近 N 次表现方差惩罚后的稳定分。
- 综合：
  - `postmortem_score = 0.6*prize_score + 0.25*structure_score + 0.15*stability_score`

## 7. 自动触发策略（复盘后）

## 7.1 触发优化条件
- 任一成立触发 `optimize_job`：
  - 最近 6 期 `postmortem_score` 均值低于阈值（默认 55）。
  - 连续 3 期无奖级命中（1-9 任意级均未命中）。
  - M4 漂移持续 `WARN/CRITICAL`。

## 7.2 触发晋升评估条件
- 仅当有新候选模型且优化成功时触发 `promotion_eval_job`。
- 评估依据：
  - M4 回测门禁结果
  - M5 最近复盘表现稳定性

## 8. 日志与审计（必须）

## 8.1 `scheduler_logs.json` 强制字段
- `task_id`, `task_type`, `trigger_source`, `status`
- `target_issue`, `snapshot_hash`, `model_version`
- `duration_ms`, `attempt_no`
- `result_summary`, `warnings[]`, `error_message`
- `created_at`

## 8.2 告警规则
- 连续 2 次 `sync_job` 失败 -> `ALERT_SYNC_DEGRADED`
- `draw_poll_job` 超过 90 分钟未命中开奖 -> `ALERT_DRAW_TIMEOUT`
- `postmortem_job` 失败 -> `ALERT_POSTMORTEM_FAILED`
- 告警必须写日志并输出可读摘要（用于后续通知渠道接入）。

## 9. API 行为规范（M5）
- `POST /api/postmortem/{issue}`
  - 入参：`issue`
  - 出参：`postmortem_id`, `score_summary`, `triggered_actions`
- `GET /api/runs`
  - 返回中新增：`postmortem_ref`, `prize_summary`
- `POST /api/sync`（M5 扩展）
  - 返回 `scheduler_context`，标识是否由定时任务触发。

## 11. 测试要求

## 11.1 单元测试
- `test_prize_rules.py`
  - 覆盖全部奖级映射路径与边界。
- `test_task_idempotency.py`
  - 同 key 重复执行必须不污染数据。
- `test_scheduler_state_machine.py`
  - 状态流转与失败重试正确。

## 11.2 集成测试
- `test_draw_to_postmortem_pipeline.py`
  - 模拟开奖后自动完成复盘与优化触发。
- `test_scheduler_logs_contract.py`
  - 校验日志字段完整性与可追溯性。

## 11.3 稳定性测试
- 模拟网络抖动/官方源失败：
  - 保证 degrade 模式不覆盖有效历史数据。
- 模拟重复触发：
  - 保证幂等不重复写入复盘与奖级记录。

## 12. 验收要求（M5 上线门槛）
- 功能验收：
  - 三类定时任务准确触发，开奖后链路自动闭环。
  - 奖级计算、命中矩阵、复盘报告正确落盘。
  - 自动优化与晋升评估按条件触发。
- 质量验收：
  - M5 新增代码覆盖率 `>= 90%`。
  - 任务幂等测试通过，重复执行不污染。
  - 日志缺字段率 `=0`。
- 可靠性验收：
  - 任务失败可重试且可追踪，失败恢复后数据一致。
  - 关键任务（sync/draw/postmortem）成功率模拟环境 `>= 99%`。
- 文档验收：
  - 产出 `artifacts/backtests/m5_acceptance_report.md`。

1. 先补任务状态机与日志契约。
2. 再实现开奖回填、奖级计算、复盘报告。
