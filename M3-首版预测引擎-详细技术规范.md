# M3 首版预测引擎详细技术规范（执行版）

## 1. 目标与原则
- 目标：交付可解释、可复现、可冻结发布的首版预测引擎。
- 原则：
  - 不使用黑箱直选。
  - 所有结果可追溯到特征、权重、搜索路径。
  - 同 `snapshot_hash + model_version + seed` 必须 100% 一致。
  - 发布结果不可回写修改（freeze）。

## 2. 范围与非范围
- 范围内：
  - 透明特征层
  - 位置评分
  - 概率校准
  - 组合搜索
  - 方案1/方案2出票
  - 发布冻结与实验 run
  - 前端 M3 展示页（质感 UI）
- 范围外：
  - 漂移降权、Optuna、冠军晋升（M4）
  - 开奖回填与复盘自动化（M5）

## 3. 技术栈（M3 固定）
- 语言：`Python 3.11+`
- 数据处理：`numpy`, `pandas`
- 模型：`scikit-learn`
- API：`FastAPI`
- 前端：`React + TypeScript + Vite`
- 图表：`ECharts`（M3 固定选型，不再二选一）
- 测试：`pytest`, `vitest`

## 4. 代码结构与职责分层
- `apps/api/app/engine/features.py`
  - 历史窗口统计、特征工程、特征标准化。
- `apps/api/app/engine/position_model.py`
  - 按位置构建评分器，输出 raw score。
- `apps/api/app/engine/calibration.py`
  - Platt scaling 拟合与推理。
- `apps/api/app/engine/search.py`
  - Beam Search + 约束评分。
- `apps/api/app/engine/ticketing.py`
  - 方案1/方案2组票。
- `apps/api/app/engine/reproducibility.py`
  - snapshot hash、seed 混合、确定性 RNG。
- `apps/api/app/services/predict_pipeline.py`
  - 串联 M3 全流程，供 `publish/predict` 调用。

## 5. 输入数据契约

## 5.1 历史数据输入
- 来源：`data/normalized/issues.json`
- 最低字段：
  - `issue`（字符串期号）
  - `front`（长度 5 的升序号码数组）
  - `back`（长度 2 的升序号码数组）
  - `draw_date`（可空）

## 5.2 快照定义（关键）
- `snapshot_hash` 计算输入：
  - 最近 `N_hist` 期标准化历史（默认 300）
  - 当前模型配置 `model_config`
  - 规则版本 `rule_version_id`
- 计算方式：
  - 对输入对象做 canonical JSON（键排序、无多余空白）
  - `sha256(canonical_json_bytes)`
- 任何输入变动都必须导致 hash 改变。

## 6. 透明特征层（硬要求）

## 6.1 号码域
- 前区：`01-35`
- 后区：`01-12`
- 对每个号码独立计算特征向量。

## 6.2 特征定义（必须全量实现）
- `freq_10`, `freq_30`, `freq_50`, `freq_100`
  - 定义：最近窗口内出现次数 / 窗口长度。
- `miss_current`
  - 定义：距离上次出现已过去的期数；未出现按窗口上限截断。
- `ewma_hotness`
  - 定义：`EWMA_t = alpha * x_t + (1-alpha) * EWMA_{t-1}`，`alpha=0.2`。
- `adjacent_last`
  - 定义：若号码与上期任一开奖号码差值为 1，则为 1，否则 0。
- `repeat_last`
  - 定义：若号码属于上期开奖号码集合，则为 1，否则 0。
- `tail_bucket`
  - 定义：尾号 `n % 10` one-hot（10 维）。
- `zone_bucket`
  - 前区分区：`[1-12], [13-24], [25-35]`（3 维 one-hot）
  - 后区分区：`[1-4], [5-8], [9-12]`（3 维 one-hot）
- `sum_contrib_proxy`
  - 定义：号码加入后对目标和值区间中心的距离改善度（归一化到 [-1, 1]）。
- `span_contrib_proxy`
  - 定义：号码加入后对目标跨度区间中心的距离改善度（归一化到 [-1, 1]）。
- `hot_cold_tag`
  - 定义：基于 `freq_30` 的 z-score；`z>0.8` 热，`z<-0.8` 冷，其余中性。
- `last_issue_interference`
  - 定义：针对上期结构过密区域加惩罚（例如连续聚集区间）。

## 6.3 特征标准化
- 数值特征使用训练窗口统计量 z-score 标准化：
  - `x'=(x-mu)/sigma`，`sigma<1e-8` 时置 0。
- one-hot 特征保持原值。
- 标准化参数写入：
  - `artifacts/backtests/feature_stats_{model_version}.json`

## 7. 位置评分模型

## 7.1 问题建模
- 前区：5 个位置模型 `front_pos_1..5`
- 后区：2 个位置模型 `back_pos_1..2`
- 对每个位置做 one-vs-all 号码打分。

## 7.2 模型算法
- 首选：`LogisticRegression(solver='lbfgs', C=1.0, max_iter=1000)`
- 样本不足回退：线性打分器（固定权重表）
- 训练数据生成：
  - 每期开奖的每个位置对应一个正样本号码
  - 同位置其余号码为负样本（可做负采样比 1:6）

## 7.3 输出
- `raw_score[number]`
- `raw_prob[number]`
- 排名前 `TopN`（默认 12 前区、6 后区）
- 每个 TopN 给出前 3 个贡献因子（绝对权重 * 特征值）

## 8. 概率校准（Platt）

## 8.1 训练策略
- 时间切分：训练集前 80%，验证集后 20%。
- 对每个位置单独拟合二元逻辑回归校准器：
  - 输入 `raw_score`
  - 标签 `hit(0/1)`

## 8.2 产出
- `calibrated_prob`
- 指标：
  - `brier_score`
  - `ece`（10 bins）
- 落盘：
  - `artifacts/backtests/calibration_{model_version}_{snapshot_hash[:8]}.json`

## 9. 组合搜索（Beam Search）

## 9.1 候选池
- 前区每位置取 top `K_front=12`
- 后区每位置取 top `K_back=6`
- 超参数进入 `model_config.search`。

## 9.2 搜索状态
- `state = {picked_front, picked_back, pos_idx, score, structure_meta}`
- 扩展时增量更新结构评分，避免全量重算。

## 9.3 硬约束（不满足即剪枝）
- 前区 5 个互异且范围 1-35
- 后区 2 个互异且范围 1-12
- 禁止极端票：
  - 前区全奇或全偶
  - 前区全小（<=17）或全大（>=18）
  - 前区长度 4+ 连号链

## 9.4 软约束评分（可调）
- `odd_even_score`
- `big_small_score`
- `zone_balance_score`
- `sum_band_score`
- `span_band_score`
- `hot_cold_mix_score`
- 合成：
  - `structure_score = Σ(w_i * score_i)`

## 9.5 多样性惩罚
- 与已入选票计算 Jaccard：
  - 前区集合相似度 > 0.8 时惩罚
  - 后区完全相同额外惩罚
- 目标：最终 10 组票不高度同质化。

## 10. 方案1 / 方案2 出票

## 10.1 方案1（5组，偏稳）
- 第 1 组：永久跟买号（Anchor Ticket）
  - 目标函数：`stability_score + high_prize_proxy - variance_penalty`
  - 默认不自动更新；仅 `POST /api/anchor/recompute` 更新。
- 第 2-5 组：结构化高分票
  - 严格结构约束 + 高位置概率优先。

## 10.2 方案2（5组，轻结构美化）
- 来源：高分池二次采样
- 规则：
  - 不允许全奇/全偶、全大/全小、极端连号
  - 结构约束低于方案1
  - 更强调“号码分布观感”
- 确定性：固定 `seed_strategy="model_version+issue+seed"`。

## 11. 可复现机制（硬门禁）

## 11.1 确定性 RNG
- 统一使用 `numpy.random.Generator(PCG64(seed_int))`
- `seed_int = hash64(snapshot_hash + model_version + str(seed))`
- 禁止使用全局随机源 `random.*` 直接采样。

## 11.2 复现记录字段
- `snapshot_hash`
- `model_version`
- `seed`
- `engine_version`
- `feature_stats_hash`
- `calibration_hash`

## 11.3 复现验收
- 同输入重复执行 20 次，输出 JSON 完全一致（包含排序顺序）。

## 12. API 详细行为（M3）

## 12.1 `POST /api/publish/{targetIssue}`
- 行为：
  - 若已有 `official` 记录，直接返回原记录（幂等）。
  - 否则执行全流程并写入 `predictions.json.official`。
- 强制字段：
  - `target_issue`, `run_id`, `model_version`, `snapshot_hash`, `seed`, `plan1`, `plan2`, `published_at`
- 错误处理：
  - 历史数据不足：`422` + `error_code="INSUFFICIENT_HISTORY"`

## 12.2 `POST /api/predict/{targetIssue}`
- 行为：
  - 每次都生成新 run（非幂等）
  - 写入 `predictions.json.experimental`
- 强制字段：
  - `run_id`, `target_issue`, `model_version`, `snapshot_hash`, `seed`
  - `plan1`, `plan2`
  - `feature_summary`, `position_summary`
  - `created_at`

## 12.3 `GET /api/analysis/{targetIssue}`
- 返回：
  - 各位置 TopN（号码、raw_score、calibrated_prob）
  - 贡献因子前 3 名
  - 结构评分分解

## 13.1 `storage/predictions.json`

- `experimental[]` 增加：

## 13.2 `artifacts/backtests/`
- `feature_stats_*.json`
- `calibration_*.json`

## 14. 前端页面质感标准（M3必须完成）

## 14.1 视觉基线
- 背景：米白纸感 + 低对比纹理 + 渐变光斑。
- 主面板：深蓝信息板 + 金属边线。
- 球体：前区红球、后区蓝球，带高光与阴影，不可扁平化。
- 字体：中文采用 `Noto Sans SC`，数字采用 `DIN` 风格字体。

## 14.2 布局
- 桌面端：
  - 左 1/3 历史滚动栏
  - 右上 1/2：模型与位置概率
  - 右下 1/2：方案票 + 实验 run + 日志
- 移动端：
  - 上下折叠
  - 保持卡片层级与留白，不得挤成纯表格。

## 14.3 动效（要克制但有质感）
- 页面加载：卡片 120ms 级联淡入。
- 球号刷新：scale + fade（150ms）。
- 图表刷新：平滑过渡，禁止瞬时闪烁替换。

## 14.4 M3 页面组件（必须）
- `PositionHeatPanel`
- `ContributionPanel`
- `PlanTicketPanel`
- `ExperimentRunPanel`
- `ModelMetaPanel`

## 15. 测试与验收清单（M3）

## 15.1 后端单测
- 特征计算正确性（至少 15 条样例）
- 位置模型输出维度与范围正确
- 校准后概率位于 `[0,1]`
- Beam Search 不产出非法票
- publish 幂等，predict 非幂等

## 15.2 复现测试
- 固定 `snapshot_hash/model_version/seed` 连续跑 20 次
- 输出哈希一致：`sha256(response_json)` 全相同

## 15.3 前端测试
- 桌面/移动布局快照测试
- 关键组件渲染测试
- 交互测试：点击实验后 run 列表新增

## 15.4 性能预算
- 单次预测（M3）目标：
  - 本地 CPU 下 `p95 < 2.5s`
- 分析接口目标：
  - `p95 < 1.2s`

## 16. 执行顺序（必须按此推进）
1. 先完成引擎内核（features/position/calibration/search）
2. 再完成 `publish/predict/analysis` 对接
3. 再补持久化字段和日志
4. 最后完成前端质感页面与测试
5. 达到 M3 全量验收后，才允许进入 M4
